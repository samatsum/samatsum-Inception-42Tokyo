# 開発者向けドキュメント - Inception インフラストラクチャ

このドキュメントは、開発者向けにインフラのアーキテクチャ、実装上の選択、およびメンテナンス手順の詳細を提供します。

## 1. 技術アーキテクチャとサービスマップ

インフラは厳格なマイクロサービス手法に従い、各サービスは独自のライフサイクルを持つ隔離されたコンテナで動作します。

### コアサービス
* **NGINX**: L7エントリーポイントおよびリバースプロキシとして機能。TLS v1.2/v1.3を終端し、パスに基づいてWordPress、Adminer、静的サイト、Grafanaへトラフィックをルーティングします。
* **WordPress**: PHP-FPM 8.2を実行。MariaDBに依存し、Docker内部DNSを通じて接続します。
* **MariaDB**: リレーショナルデータベース。内部ブリッジネットワーク内からの接続のみを許可するように設定されています。

### ボーナスサービス
* **Redis**: WordPress用インメモリキャッシュ。DB読み取りの時間計算量をO(1)に低減します。
* **Adminer**: MariaDB管理用GUI。
* **Prometheus**: メトリクス収集用時系列データベース（TSDB）。
* **Grafana**: Prometheusデータの可視化ダッシュボード。
* **Static-Site**: Python/FlaskベースのステートレスWebアプリケーション。
* **FTP Server**: WordPressボリューム上のファイルを直接操作するための `vsftpd`。

## 2. 実装の詳細

### PID 1 とシグナルハンドリング
グレースフルシャットダウン（終了コード0）を確実にするため、すべてのエントリーポイントスクリプトは `exec` コマンドを使用してシェルプロセスをサービスデーモンに置き換えます。これにより、デーモンがDockerエンジンからSIGTERMを直接受信できるようになります。

### 初期化の冪等性（Idempotency）
各スクリプトは初期化前に現在の状態を確認します：
* **MariaDB**: `mysql_install_db` を実行する前に `/var/lib/mysql/mysql` が存在するか確認します。
* **WordPress**: 設定コマンドを実行する前に `wp-config.php` の存在を確認します。

### レースコンディションの回避
WordPress初期化スクリプトは、`wp db check` を用いてMariaDBに対するL7ヘルスチェックをリトライループ内で実行します。これにより、WordPressがインストールを試みる前に、データベースがクエリを受け付けられる状態であることを保証します。

## 3. ストレージと永続化

課題の要件に従い、特定のホストパスにマップされた **Docker 名前付きボリューム** を使用しています：
* **Database**: `/home/samatsum/data/mariadb`
* **WordPress**: `/home/samatsum/data/wordpress`

`Makefile` はコンテナが起動する前に、これらのディレクトリを正しい権限（`755`）で作成することを保証し、Dockerがroot所有のディレクトリとして自動作成するのを防ぎます。

## 4. セキュリティ

* **Secrets**: すべてのパスワードは Docker Secrets で管理され、`/run/secrets/` 内に読み取り専用ファイルとしてマウントされます。
* **ログ保護**: シェルスクリプトは機密変数を展開する前に `set +x` を実行し、デバッグ中にパスワードがログに漏洩するのを防ぎます。
