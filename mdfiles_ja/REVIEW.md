# プロジェクト評価ガイドライン

## 予備
✅ [ORIGINAL START]

* 不正行為が疑われる場合は、評価はここで終了します。「不正行為」フラグを使用して報告してください。冷静かつ賢明に判断し、このボタンの使用には十分ご注意ください。
* 予備テスト
* 例えば、ローカルの .env ファイルを使用して情報を保存したり、Docker のシークレットを使用したりすることは許可されています。評価中に作成されたシークレット ファイル以外で、git リポジトリに認証情報、API キー、またはパスワードが存在する場合、評価は停止し、マークは 0 になります。
* 評価対象となる学習者またはグループが出席している場合にのみ、評価は実施されます。こうすることで、全員が知識を共有し、共に学ぶことができます。
* 提出物が全くない場合（またはファイル、ディレクトリ、ファイル名が間違っている場合）、成績は0となり、評価プロセスは終了します。
* このプロジェクトでは、彼らのGitリポジトリを彼らの端末にクローンする必要があります。

✅ [ORIGINAL END]

* **検証手順 (機密情報の漏洩チェック)**: 
  `cat .gitignore` を実行し、パスワードファイルがGit管理から除外されていることを見せる。
* **【解説】堅牢性の担保**: 
  `.env` にパスワードを直接書かず、Docker Secrets を用いることで、コンテナ内の `/run/secrets/` に `tmpfs` (RAM上の一時ファイルシステム) としてマウントされる。物理ディスクに一切書き込まれないため、極めて高いメモリ安全性が担保される。

## 一般的な指示
✅ [ORIGINAL START]

* 評価プロセス全体を通して、要件の確認方法や検証方法がわからない場合は、評価対象の学習者に協力してもらう必要があります。
* アプリケーションの設定に必要なすべてのファイルがsrcsフォルダ内に配置されていることを確認してください。srcsフォルダはリポジトリのルートディレクトリに配置する必要があります。
* リポジトリのルートディレクトリにMakefileが存在することを確認してください。
* 評価を開始する前に、ターミナルで次のコマンドを実行してください。

```bash
docker stop $(docker ps -qa); docker rm $(docker ps -qa); docker rmi -f $(docker images -qa); docker volume rm $(docker volume ls -q); docker network rm $(docker network ls -q) 2>/dev/null

```

* docker-compose.yml ファイルを読み込んでください。「network: host」や「links:」が含まれていてはいけません。含まれている場合は、評価はここで終了します。
* docker-compose.yml ファイルを読み込んでください。必ず「network(s)」という記述が必要です。記述がない場合、評価はここで終了します。
* Makefileと、Dockerが使用されているすべてのスクリプトを調べてください。いずれのスクリプトにも「--link」が含まれていてはなりません。含まれている場合は、評価はここで終了します。
* Dockerfileを調べてください。ENTRYPOINTセクションに「tail -f」やバックグラウンドで実行されるコマンドが含まれている場合、評価はここで終了します。「bash」または「sh」が使用されている場合でも、スクリプトの実行に使用されていない場合（例：「nginx & bash」または「bash」）も同様です。
* Dockerfileを確認してください。コンテナは、AlpineまたはDebianの最新安定版のいずれかをベースに構築する必要があります。
* エントリポイントがスクリプトの場合（例：ENTRYPOINT ["sh", "my_entrypoint.sh"]、ENTRYPOINT ["bash", "my_entrypoint.sh"]）、バックグラウンドでプログラム（例：'nginx & bash'）が実行されないようにしてください。
* リポジトリ内のすべてのスクリプトを調べてください。無限ループを実行するスクリプトがないことを確認してください。禁止されているコマンドの例をいくつか示します。「sleep infinity」、「tail -f /dev/null」、「tail -f /dev/random」
* Makefileを実行します。

✅ [ORIGINAL END]

* **検証手順 (クリーンアップ)**:
  `make fclean` または提示された `docker rm` コマンド等で環境を初期化。
* **【解説】禁止事項の原理**:
  * **`network: host` が禁止される理由**: L3ネットワークの隔離（Network Namespace）を無効化し、ホストマシンのNICを直接共有してしまうため、コンテナ間の境界やセキュリティが崩壊する。
  * **`tail -f` 等のバックグラウンド実行が禁止される理由**: Dockerコンテナは「PID 1」の主プロセスが終了すると停止する。`tail -f` でごまかすのではなく、NGINXなら `exec nginx -g "daemon off;"` のようにフォアグラウンドプロセスとして起動させ、OSからのシグナル（SIGTERM等）を正しく受け取れる設計にする必要がある。

### 活動概要
✅ [ORIGINAL START]

評価対象の学習者は、以下のことを分かりやすい言葉で説明する必要があります。

* DockerとDocker Composeの仕組み
* Docker Compose を使用した場合と使用しない場合の Docker イメージの違い
* 仮想マシンと比較したDockerの利点
* 本プロジェクトに必要なディレクトリ構造の妥当性（例は主題のPDFファイルに記載されています）。

✅ [ORIGINAL END]

* **DockerとDocker Composeの仕組み (なぜ動くのか)**:
  * **Docker**: Linuxカーネルの標準機能である `Namespaces` (プロセス番号やネットワーク等の空間隔離) と `cgroups` (CPU・メモリ等のリソース制限) を組み合わせたプロセス隔離技術。
  * **Docker Compose**: 「理想のシステム状態」と「現状」の差分を埋める宣言的オーケストレーションツール。L3ネットワーク（Bridge）を一括作成し、内部DNSによる名前解決（`wordpress:9000` 等での通信）を可能にする。

* **Docker Compose を使用した場合と使用しない場合の Docker イメージの違い**:
  * **違いはない**。イメージそのもの（バイナリや OverlayFS レイヤー）は同一。違うのは「実行時のコンテキスト（環境変数、ネットワーク、マウントの引数等）」。Composeを使うことで環境構築がIaC (Infrastructure as Code) として固定化され、再現性と可読性が飛躍的に向上する。
* **仮想マシン(VM)と比較したDockerの利点**:
  * VMは「ゲストOSごとハードウェアをエミュレートする」ため時間・空間計算量のオーバーヘッドが大きいが、Dockerは「ホストOSのカーネルを共有する、隔離されたただのプロセス」であるため、起動がミリ秒単位でリソース効率が極めて高い。

### Dockerの基本
✅ [ORIGINAL START]

* まず、Dockerfileを確認してください。サービスごとに1つのDockerfileが必要です。Dockerfileが空ファイルでないことを確認してください。そうでない場合、またはDockerfileが欠落している場合は、評価プロセスはここで終了します。

✅ [ORIGINAL END]

### WordPressとphp-fpm、そしてそのボリューム
✅ [ORIGINAL START]

* Dockerfileが存在することを確認してください。
* DockerfileにNGINXが含まれていないことを確認してください。

* 管理ダッシュボードからページを編集します。ウェブサイト上でページが更新されていることを確認してください。

✅ [ORIGINAL END]

* **検証手順 (ボリュームと動作)**:
  * `docker volume ls` -> `docker volume inspect <volume_name>` を実行し、出力の `Options.device` にホストパス (`/home/login/data/`) が含まれていることを見せる。
  * WordPress管理者画面へログイン（ユーザー名にadminが含まれていないことをアピール）。ページを編集・更新する。

  NGINX自身はPHPを解釈できないため、HTTPリクエストを FastCGI プロトコルに変換し、9000番ポート経由で専門のプロセス（PHP-FPM）へ委譲するアーキテクチャとなっている。

### 粘り強さ！
✅ [ORIGINAL START]

* この部分は非常に簡単です。仮想マシンを再起動する必要があります。再起動後、docker compose を再度起動します。次に、すべてが正常に動作していること、および WordPress と MariaDB の両方が正しく設定されていることを確認します。以前に WordPress ウェブサイトに加えた変更が反映されているはずです。上記のいずれかの点が正しくない場合は、評価プロセスはここで終了します。

✅ [ORIGINAL END]

* **検証手順 (Persistence テスト)**:
  1. WordPress上で記事やコメントを変更する。
  2. `sudo reboot` (または `sudo make stop`) でシステムをダウンさせる。
  3. `sudo make up` (または `make start`) で復帰させる。
  4. 再度ブラウザ等でアクセスし、変更内容が維持されていることを証明する。
* **【解説】バインドマウントの利点**:
  ホストマシンのディレクトリをコンテナ内に直結（バインドマウント）させているため、コンテナプロセスが破棄されても、ディスクI/Oとしてホスト側に書き込まれたデータは消滅しない。

## ボーナス
✅ [ORIGINAL START]

このプロジェクトでは、ボーナスパートはシンプルな内容となっています。追加するサービスごとにDockerfileを作成する必要があります。各サービスは独自のコンテナ内で実行され、必要に応じて専用のボリュームが割り当てられます。ボーナスパートは、必須パートが完全に完了している場合にのみ評価されます。「完璧」とは、必須パートが完全に完了し、不具合なく動作することを意味します。必須要件をすべて満たしていない場合、ボーナスパートは一切評価されません。

**VM:** ゲストOSを丸ごと起動するため、数GBのメモリと数分の起動時間を消費する 。
**Docker:** ゲストOSを持たず、ホストOSのカーネルを共有してプロセスを隔離 (Namespace/Cgroups) しているだけである 。起動時間はミリ秒単位で、空間計算量も極めて少ない 。
**【根本原理】** VMが「物理マシンをソフトウェアで再現したもの」であるのに対し、Dockerは「ホストOS上で動く、隔離されたただのプロセス」である 。


 
**Namespaces (名前空間):** プロセスが見えるリソース（プロセスID、ネットワーク等）を切り離し、「自分専用のOSで動いている」と錯覚させる 。
**cgroups:** コンテナが使用できるハードウェアリソースを制限・監視し、暴走を防ぐ

**マウントの概念 (ハイブリッド方式):** `driver: local` で `device` を指定することで、実態はホストディレクトリ直結の「バインドマウント」でありながら、Docker側には「名前付きボリューム」として登録され、管理が容易になる 。


**Docker Secrets の堅牢性:** ホスト上のファイルを読み込み、コンテナ内部の `/run/secrets/` に `tmpfs` (RAM上の一時ファイルシステム) を作成してマウントする 。ディスクに一切書き込まれないため、極めて高いメモリ安全・堅牢性を担保する 。





# アーキテクチャ設計の比較解説

Inceptionプロジェクトにおける技術的選択の正当性を説明するための、4つの重要なアーキテクチャ比較です。

---

## 1. Virtual Machines (仮想マシン) vs Docker

両者は「システムを隔離して動かす」という目的は同じですが、仮想化するレイヤーが根本的に異なります。

| 比較項目 | Virtual Machine (VM) | Docker (コンテナ) |
| :--- | :--- | :--- |
| **仮想化レイヤー** | ハードウェアレベル (Hypervisorを介する) | OSレベル (Linuxカーネルを共有) |
| **ゲストOS** | 独立したゲストOSを丸ごと起動する | なし (ホストOSのカーネルをそのまま使う) |
| **空間計算量 (容量)** | 数GB〜数十GB (OSのシステムファイルを含むため重い) | 数MB〜数百MB (アプリと依存ライブラリのみ) |
| **時間計算量 (起動)** | 数分単位 (OSのブートプロセスが必要) | ミリ秒〜数秒単位 (単なるプロセスの起動) |
| **隔離の仕組み** | ハードウェアの完全なエミュレーション | Namespaces (空間隔離) と cgroups (リソース制限) |

**【結論】**
VMは「物理マシンをソフトウェアで再現したもの」であり、完全な隔離を提供しますがオーバーヘッドが甚大です。一方Dockerは「ホストOS上で動く、隔離されたただのプロセス」です。カーネルを共有しているため無駄なリソース消費がなく、極めて高い効率性と可搬性を実現します。

---

## 2. Secrets vs Environment Variables

パスワードやAPIキーなどの機密情報（クレデンシャル）をコンテナに渡す手法の比較です。

| 比較項目 | Environment Variables (環境変数) | Docker Secrets (シークレット) |
| :--- | :--- | :--- |
| **データの受け渡し** | コンテナ起動時の環境変数（ENV）として注入 | 特定のディレクトリにファイルとしてマウント |
| **視認性 (漏洩リスク)** | `docker inspect` コマンドやプロセスの環境変数一覧から**平文で丸見え**になる | ファイルを読み取る権限を持つプロセスからしか見えない |
| **ディスクへの書き込み**| ホストやコンテナのディスク上に平文で残るリスクが高い | `tmpfs` (RAM上の一時ファイルシステム) に展開されるため、**物理ディスクに一切書き込まれない** |
| **更新の容易さ** | 変更するにはコンテナの再作成が必要 | 動的な更新やバージョン管理がシステム的に容易 |

**【結論】**
環境変数は設定値（ドメイン名やポート番号など）を渡すには便利ですが、機密情報の保存には適していません。Docker Secretsを使用することで、データがメモリ上（tmpfs）のみに保持され、極めて高い堅牢性とメモリ安全性が担保されます。

---

## 3. Docker Network vs Host Network

コンテナの通信経路（L3/L4ネットワーク）をどのように隔離するかの比較です。

| 比較項目 | Host Network (`network: host`) | Docker Network (Bridge等) |
| :--- | :--- | :--- |
| **ネットワークの共有** | ホストマシンのNICをそのまま直接共有する | 独立した仮想L3スイッチ（ブリッジ）を作成する |
| **ポートの競合** | コンテナが80番を使えば、ホストの80番が直接塞がる | 内部では同じポートを使っても、ホスト側でマッピング（`443:8443`等）を変えれば競合しない |
| **セキュリティ (隔離性)**| **ゼロ**。ホストと全く同じネットワーク空間に晒される | 仮想ルーターによって**完全に隔離**される。明示的にポートを開けない限り外部からアクセス不可 |
| **名前解決 (DNS)** | ホストの `/etc/hosts` 等に依存 | Docker内部のDNSサーバーにより、`wordpress` 等のコンテナ名で相互通信が可能 |

**【結論】**
Host Networkはネットワークの隔離（Network Namespace）を破壊し、セキュリティ上の大きな脆弱性を生みます。Docker Network（Bridge）を使用することで、L3レイヤーでのコンテナ間の安全な通信（内部DNS）と、L4レイヤーでの厳密なアクセス制御（必要なポートだけを外に開く）が可能になります。

---

## 4. Docker Volumes vs Bind Mounts

コンテナが破棄されてもデータを失わないための「データの永続化」手法の比較です。

| 比較項目 | Bind Mounts (バインドマウント) | Docker Volumes (名前付きボリューム) |
| :--- | :--- | :--- |
| **保存場所の指定** | ホストOSの**絶対パス**を直接指定する (例: `/home/user/data`) | Docker Engineが管理する領域 (例: `/var/lib/docker/volumes/`) に保存される |
| **管理の主体** | ホストOSのファイルシステム | Dockerデーモン |
| **依存性** | ホストのディレクトリ構造やパーミッション（権限）に強く依存するため、別のPCに持っていくと動かないことがある | Dockerがよしなに管理するため、ホストの環境に依存せず移植性が高い |
| **機能性** | 単純なディレクトリの直結 | ボリュームドライバを利用して、クラウドストレージに直接繋ぐなどの拡張が可能 |

**【結論】**
Bind Mountsは手元のコードをコンテナに同期させるような開発環境では便利ですが、ホスト環境への依存度が強くなります。Docker Volumesを使用することで、データの管理をDockerに一任でき、システムの堅牢性と移植性が向上します。
*(※Inception課題では要件により、Volumesの仕組みを使いながら実態はホストの特定パスに結びつけるハイブリッドな設定 `driver_opts: device` を用いて永続化を実現します)*
