README.md (和訳)

このプロジェクトは42のカリキュラムの一環として samatsum によって作成されました。

1. 概要
Docker Composeを用いて、マルチサービス・インフラを仮想化するシステム管理プロジェクトです。このアーキテクチャは厳格なマイクロサービス手法に従い、隔離、セキュリティ（TLS v1.2/v1.3）、および自己修復設定による高可用性を確保しています。

2. セットアップの概要
インフラをデプロイするには以下の手順を実行してください（詳細な環境準備については DEV_DOC.md を参照）：

    secrets/ ディレクトリと .env ファイルを準備。

    hostsファイルで samatsum.42.fr を 127.0.0.1 にマッピング。

    ルートディレクトリから make up を実行。
    サービスの使用方法や管理コマンドについては USER_DOC.md を参照してください。

3. 設計の選択と比較（Design Choices）

    VM vs Docker: VMはハードウェア層を仮想化するため、インスタンスごとに完全なゲストOSを必要とし、空間的・時間的コストが肥大化します。DockerはOSカーネルレベルで仮想化し、ホストカーネルを共有することでリソース消費を最小化し、ほぼ瞬時の起動（プロセス生成 O(1)）を可能にします。

    Secrets vs 環境変数: 環境変数は docker inspect 等で外部から閲覧可能ですが、Docker Secretsはメモリ上の tmpfs にマウントされるため、機密情報（パスワード）がコンテナのディスクに触れることなく、極めて高いセキュリティを維持できます。

    Docker Network vs Host Network: ホストネットワークは隔離性を排除しますが、ブリッジネットワークを使用することでL3層の隔離を実現し、コンテナ間通信をサービス名による内部DNSに限定。外部への露出をNGINXの443ポートのみに絞り、最小権限の原則を徹底しています。

    ボリューム vs バインドマウント: バインドマウントはホストパスに依存しポータビリティを損なうため、ローカルドライバを用いた名前付きボリュームを採用。Dockerの管理サイクルを活用しつつ、指定されたパス（/home/samatsum/data）への永続化要件を両立させています。

    AIの利用: インフラ構成の監査、L7プロトコル（Prometheus/NGINX）のミスマッチのデバッグ、およびPID 1のシグナル伝搬（execパターン）の最適化において、技術メンターとしてAI（Claude）を活用しました。すべてのロジックは手動で検証済みです
